Date: 2026-05-13
Project: War Buddy (Godot 4.6.x)

## 1. Goal

Land the smallest visible piece of the P0 chain from the audit: a graybox world that contains **both faction main bases with starting resources**, so that the LLM deputy's `BattlefieldSnapshotBuilder` for the first time observes "real RTS state" (8 mineral patches, 1 gas geyser, 1 HQ, 6 workers, 1 hero per faction) instead of the current "3 enemy buildings + 1 hero".

v0.15.0 ships **placement and identity only**. No gathering, no construction, no production. Workers stand idle around the HQ; resources sit visible but inert. That keeps the diff focused and lets v0.15.1 / v0.15.2 layer on terrain + behavior without simultaneous churn.

## 3. Open Implementation Questions (Resolved Inline)

### 3.1 Single-faction vs two-faction state

09 §11 supports `GameState.all_factions()` returning multiple. Currently only `player` is seeded. v0.15.0 enemy HQ exists in-scene but **does not** enter `GameState.alive_buildings` of an `enemy` faction — it remains a "scene-only" entity, queried by `EnemyBuilding.gd`'s existing path that uses scene-tree groups (`enemy_buildings`).

Rationale: minimal scope. The LLM snapshot already reads enemies via `BattlefieldSnapshotBuilder._enemy_buildings()` which uses scene groups. Adding a full enemy faction record duplicates state without buying anything in v0.15.0. v0.16 wires the enemy faction proper when production lands.

### 3.2 Workers in `squad_units` group

Earlier brainstorm Q5 = A+C: workers join `squad_units`. Side effects to verify:

- `EnemyBuilding`'s defensive attack iterates `squad_units` — workers will be targeted in range. Acceptable (workers are valid combat targets per 09 §3.3 — they have HP 45 / armor 0 / light).
- `DevSquadController` K-key damage will hit workers. Acceptable for debug.
- `BattlefieldSnapshotBuilder._friendly_units()` already reads `squad_units` — workers automatically appear in deputy's observation.

## 4. Task Breakdown

### 4.2 `PlayerHq.gd` + `hq.tscn` (`v0.15.0-2`)
- New file: `godot/scripts/player_hq.gd`
  - `extends StaticBody3D`
  - `@export var faction_id: StringName = &"player"`
  - `@export var building_def_id: StringName = &"hq"`
  - On `_ready`: load `BuildingDef` from `EntityLibrary.building(building_def_id)`; join `friendly_structures` group; register with `FactionState.alive_buildings` via `GameState.register_completed_building(faction_id, building_def_id)` (which also bumps `supply_max` by HQ's `supply_provided=10`)
- New file: `godot/scenes/hq.tscn`
  - Root: `StaticBody3D` (script = `player_hq.gd` initially; will refactor for enemy variant in step 4.3)
  - 4×4×4 `BoxMesh` with blue tint material
  - `CollisionShape3D` matching the box
  - Sized for `friendly_structure` physics layer + `cursor_pick`

### 4.3 Enemy HQ variant
- Refactor `hq.tscn` to support faction variant:
  - Single scene, `@export var is_enemy_hq: bool = false` on the script
  - On `_ready`, if `is_enemy_hq`: swap material to red, join `enemy_buildings` group, attach `EnemyBuilding.gd` behavior (defensive attack + destruction → victory check)
  - If not enemy: behavior per 4.2
  - Concrete implementation: `hq.tscn` carries a `script` that delegates — use a thin `Hq.gd` dispatcher that swaps in `PlayerHq.gd` or `EnemyBuilding.gd` behavior based on the export. (Or: two scenes inheriting a common base — simpler in Godot.)
- Decision: **two scenes** `player_hq.tscn` + `enemy_hq.tscn`, each instancing a common `hq_body.tscn` for the mesh/collision. Saves the if-branch in script.
- Place `enemy_hq.tscn` at `(22, 0, -22)` in `world.tscn`

### 4.4 `worker.tscn` + 6 starting workers

### 4.5 `ResourceNode.gd` + `mineral_patch.tscn` + `gas_geyser.tscn`
- New file: `godot/scripts/state/resource_node_body.gd`
  - `extends StaticBody3D`
  - `@export var node_id: StringName`
  - `@export var resource_type: StringName  # "mineral" or "gas"`
  - `@export var initial_amount: int = 1500  # 1500 for mineral, 0 for gas (cycle-only)`
  - On `_ready`: instantiate a `ResourceNodeDef` (existing class from v0.10.1) with the export values; register with `GameState.resource_nodes(...)` (which becomes a real query method in this commit); join `resource_nodes_mineral` or `resource_nodes_gas` group
- `GameState.resource_nodes(resource_type)` — implement as scene-group query (returns Array)
- New file: `godot/scenes/mineral_patch.tscn`
  - `StaticBody3D` + `BoxMesh` (1.5×1×1.5 m, cyan-blue) + `CollisionShape3D`
  - Script = `resource_node_body.gd`; `resource_type = &"mineral"`; `initial_amount = 1500`
- New file: `godot/scenes/gas_geyser.tscn`
  - Same as mineral_patch but `CylinderMesh` (radius 1, height 3, green); `resource_type = &"gas"`; `initial_amount = 0` (`depletes = false`)
- Place 8 mineral + 1 gas instances at player main per §3.3 coordinates
- Place mirror 8 mineral + 1 gas at enemy main

## 6. Risk and Rollback

- **Risk:** the existing `EnemyBuilding.gd` defensive-attack iterating `squad_units` might focus-fire on workers from too far away if our enemy HQ has the default `attack_range = 6m`. Mitigation: enemy HQ is at NE corner, workers at SW — ~50m apart, well out of range. No change needed.
- **Risk:** scaling the nav-mesh region may require regenerating bake. Mitigation: trigger nav-bake in editor before commit.
- **Risk:** existing tests reference `EnemyBuildingA/B/C` by name or count. Mitigation: §4.8 includes a sweep — if any v0.14.x test fails, update its assertions to match the new world.

Rollback: revert the 9 task commits in reverse order; map and scene state returns to v0.14.0.
