# MVP 物理与交互手感规范

日期：2026-04-27
项目：War Buddy（Godot 4.6.x）
状态：草案。范围限定于当前 MVP 灰盒（`godot/scenes/main.tscn`）。独立于 MVP 之后文档线（06–10、12），但与之保持一致。

> 本文为 [`11-mvp-physics-and-feel.md`](11-mvp-physics-and-feel.md) 的中文参考版。两份内容保持同步；如发生冲突，以英文版为权威。

## 1. 目的

把 MVP 的**物理交互层**钉死，让它能朝一个明确的手感目标调参，而不是漂移。

两个锚点：

- **MVP 手感目标 = "B"（LOL / DOTA 响应优先）**——输入立即兑现、动作直接、反馈极简但明确。
- **远期手感目标 = "D"（动作 RPG 现代综合）**——数值和架构留升级空间，让 MVP 数值能往更丰富的动作手感方向调，而不用重写系统。

本文每条规范都是**可量化数值 + 主观意图**。数值是起点，由 playtest 调；主观意图是契约——如果调参保持了数值却丢了意图，那要改的是数值。

## 2. 碰撞与图层矩阵

### 2.1 物理图层

`project.godot` 中分配下列 Godot 物理图层（具体编号待定）：

| 图层 | 成员 | 作用 |
|---|---|---|
| `terrain` | 静态世界几何、NavObstacle | 对所有人都是硬墙 |
| `hero` | 玩家英雄 | 特殊优先级实体 |
| `friendly_unit` | 未来的己方小队长、普通兵 | 互相软碰撞，与英雄软碰撞 |
| `enemy_unit` | 未来的敌方单位 | 与英雄硬碰撞 |
| `friendly_structure` | 未来的己方建筑 + 施工 ghost | 硬墙、寻路阻挡 |
| `enemy_structure` | 敌方建筑（当前 MVP 目标） | 硬墙、寻路阻挡 |
| `corpse` | 死亡单位的 ragdoll 尸体 | 软碰撞、可推 |
| `soul` | 英雄 / 小队长死亡的漂浮灵魂 VFX（副官在场外，不参与） | 无碰撞 |
| `attack_hitbox_player` | 己方触发的近战/弹道命中体 | 穿过自己一方 |
| `attack_hitbox_enemy` | 敌方攻击命中体 | 检测英雄但不阻挡移动 |
| `cursor_pick` | 鼠标射线可拾取目标 | 高优先级选中 |

### 2.2 英雄碰撞矩阵

| 对方实体 | 行为 | 备注 |
|---|---|---|
| 敌方建筑 | **硬碰撞 + 寻路阻挡** | MVP 已实现 |
| 敌方单位（未来） | **硬碰撞** | 拦截、堵口、肉盾有意义 |
| 己方单位（未来） | **软碰撞** | 互推不卡死 |
| 己方建筑（未来） | **硬碰撞 + 寻路阻挡** | 施工 ghost 从摆放即生效（见 §2.4） |
| 地形 / 不可走区 | **硬碰撞 + 寻路阻挡** | 包括悬崖、河流、设计师墙体 |
| 己方攻击 hitbox | **穿过** | 英雄不会被自己的武器命中 |
| 敌方攻击 hitbox | **检测、不阻挡** | 造成伤害；不影响移动 |
| 鼠标 picking 射线 | **可选中、高优先级** | 重叠时英雄优先 |

### 2.3 己方推挤优先级

己方单位之间软碰撞时的推挤求解：

> **英雄拥有无敌优先级。其它己方永远让位英雄。**

**主观意图**：强化"主角感"。英雄永远不需要等部队让路；部队会为英雄分开。

**实现要点**：己方单位与英雄发生碰撞时，物理体使用更低质量 / 更高被推系数，与速度状态无关。

### 2.4 施工 ghost

己方建筑摆放时（ghost / 施工中状态），**自摆放瞬间起**即成为 `NavigationObstacle`，不等到完工。英雄从第一帧起就绕开 ghost 寻路。

**主观意图**：防止玩家误穿过自己的施工地点，被自己造的建筑包死。

## 3. 死亡与尸体处理

所有**场上**单位死亡都留下 **ragdoll 尸体**，"Human Fall Flat 果冻物理"风格。尸体在 `corpse` 图层：软碰撞、可被任何移动单位推开、不阻挡寻路。

此外，`agency_tier ∈ {hero, captain}` 的场上单位（见 06 §2.3 与 09 的 `agency_tier` 字段）死亡时额外生成**漂浮灵魂** VFX，在 `soul` 图层：无碰撞、向上漂浮、数秒内淡出。

**副官不在场上**（06 §2.3），因此完全不参与尸体 / 灵魂管线。它没有碰撞体、没有 HP 条、没有死亡状态。局内表现形式是 HUD 头像 + 语音——视觉处理详见 06 §8 开放问题，延后到后续 UX 文档。

| agency_tier | 是否在场上 | Ragdoll 尸体 | 漂浮灵魂 |
|---|---|---|---|
| hero | 是 | 是 | 是 |
| deputy | **否** | n/a | n/a |
| captain | 是 | 是 | 是 |
| regular | 是 | 是 | 否 |

**主观意图**：

- Ragdoll 给整个游戏注入一层"物理喜剧"暗调，与战略指挥的严肃感形成反差，情感可读，不冷冰冰。
- 灵魂区分了玩家有羁绊（或可能产生羁绊）的单位。小队长死亡的视觉份量与普通兵不同。

## 4. 英雄移动规范

下列数值都是**目标值，由 playtest 调**。可调参数列表见 §10。

### 4.1 最大速度（Q1）

- **目标**：英雄无障碍下满速横穿地图对角线**约 45 秒**。
- **推导**：`hero_max_speed = map_diagonal_meters / 45 s`。当前 MVP 灰盒地图测量待定，暂估 2.0–2.5 m/s。
- **主观意图**：介于 DOTA（慢、慎重）和老 RTS（迟缓）之间。地图应该感觉像战场，不是走廊。

### 4.2 加速度（Q2）

- **目标**：0 → 最大速度 **80–120 ms**（起点 100 ms）。
- **推导**：`accel ≈ max_speed / 0.1 s`。
- **主观意图**：输入立即被兑现，但保留一丝质量感。如果英雄感觉"漂浮 / 气垫船"——加速度太高；如果感觉"踩泥巴"——太低。

### 4.3 转向（Q3）

- **逻辑**：朝向方向**立即**跟随输入。同帧内移动方向变更。
- **视觉**：网格的可视旋转用 **~100 ms** 缓动到新逻辑朝向。
- **主观意图**：玩家永远不会感觉转身动画在拖累自己的输入。动画是礼貌，不是闸门。这是"B 派加抛光层"的选择。

### 4.4 急停（Q4）

- **目标**：停止命令或路径终点时**立即**减速到 0（0 ms，单帧速度归零）。
- **主观意图**：玩家叫停，英雄就停。无滑步、无超调。配合 §4.2 的轻微启动加速，"启动慢、停止硬"的不对称给出"反应灵敏但有质感"的辨识签名，与 DOTA（对称缓动）和纯街机（双向瞬时）都不同。

### 4.5 移动输入模型

当前 MVP 使用**左键**作为主命令输入（与 `hero_controller.gd` 一致）。B 派标杆 LoL/DOTA 的规范是右键；按钮迁移作为未来可选项，不在 11 范围内（见 §11）。

- 左键点击地面 → 移动到该点，沿障碍物寻路。
- 左键点击敌方 → 进入射程并自动攻击。
- 左键点击不可达点 → 走到同一 NavMesh 岛上**最近可达点**（Q9 = c，见 §6）。

## 5. 摄像机

### 5.1 默认行为（Q12）

- 摄像机默认跟随英雄。
- 玩家可随时通过 §5.3 的控制方式自由 pan 走。
- 按**空格键**（未来：双击英雄头像 HUD）重新居中并锁定到英雄。
- 跟随脱离阈值：pan 偏移超过视口对角线 1.5 倍后视为脱离（可调）。

**主观意图**：像 LoL 一样以英雄为中心，但玩家想看哪就看哪，不需要跟摄像机较劲。

### 5.2 视角与投影（Q14）

- 高俯视、近正交。Pitch ~70–80°，轻微透视。
- **主观意图**：经典 RTS 可读性。尽管 06 写"指挥官在场"，摄像机视角仍然战略——*单位*在场，不是*摄像机*在场。

### 5.3 控制方式（Q13）

- **屏幕边缘 edge-scroll**（速度可调，提供"关闭"开关给讨厌它的玩家）。
- **WASD / 方向键**键盘平移。
- 鼠标滚轮缩放（限定范围内）。
- 左键是主命令按钮（见 §4.5），不用于摄像机拖拽。
- 右键和中键 MVP 不用。右键预留给未来迁移到 B 派规范或上下文操作；中键预留给未来的地图 ping / 快速居中。

## 6. 点击与目标反馈

### 6.1 地面点击（Q7 = a）

- **视觉**：光标本身播放短暂涟漪/脉冲。**不在地面绘制 ping 圆环或持续旗帜**。
- **行为**：移动命令在点击瞬间立即生效。
- **主观意图**：英雄开始移动**就是**反馈。极简强化"游戏信任你的意图"的产品哲学。

### 6.2 鼠标悬停在敌方（Q8 = b）

- 被悬停的敌方单位/建筑脚下出现红色环。
- 环淡入：≤ 80 ms。
- 移开悬停时移除。
- **主观意图**：清晰的目标提示，不需要"全身轮廓发光"那种视觉噪音。

### 6.3 不可达点击（Q9 = c）

- 无错误音效，无拒绝视觉。
- 英雄寻路到最近可达点。
- **主观意图**：游戏信任玩家的意图并尽力执行，而不是因为玩家点错了就训斥。和 §6.1 极简哲学一致。

## 7. 战斗反馈

### 7.1 Hitstop（Q5 = b）

- 近战命中落点：双方动画/移动**冻结 30–60 ms**（起点 45 ms）。
- 弹道：仅命中受害方 hitstop。
- **主观意图**：有重量感但不打断节奏。玩家应该感觉到打击，但不要让流程被打断。

### 7.2 屏幕震动（Q6 = b）

- 仅在**重要事件**触发：
  - 英雄被造成超过阈值的伤害（如单次 > 10% 最大 HP）
  - 英雄确认击杀（last-hit 反馈）
  - 己方建筑被摧毁
  - 敌方建筑被摧毁（胜利反馈）
- 振幅：克制。按事件严重度缩放。普通伤害交换不触发。
- **主观意图**：震屏稀有才有意义。如果一直震，注意力调度功能就消失了，只剩噪音。

### 7.3 HP 条响应

- HP 条（单位上方 + 英雄 HUD）双层动画：红色立即下降，白色"幽灵条"延迟约 400 ms 追平。
- **主观意图**：玩家能立即看到*被打了*，稍后看到*被打了多少*——MOBA 标准语言。

## 8. 寻路边界情况

### 8.1 英雄被推出 NavMesh（Q10 = b）

- 每帧检查英雄距最近 NavMesh 顶点的距离。
- 偏移超过 **1.5 m** 持续 **3 帧**以上，传送到最近合法 NavMesh 点。
- **主观意图**：硬保底防"卡进几何"bug。3 帧缓冲避免合法 ragdoll 推挤事件触发误判。

### 8.2 目标在孤岛（Q11 = b）

- 目标点不可达（不同 NavMesh 岛、完全封闭）：
- 英雄寻路到自己所在岛上**直线距离最近**目标点的位置。
- 无错误反馈。同 §6.3 哲学。

### 8.3 寻路完全失败（Godot 返回空路径，本岛上也找不到路）

- 英雄保持原地不动。
- 控制台输出警告（开发者面，不暴露给玩家）。
- 此情况在正确制作的地图中不应出现；日志用于尽早暴露关卡作者 bug。

## 9. 尸体推挤求解

- 活着的单位从尸体层穿过：尸体获得推力速度，~1.5 s 内阻尼归零。
- 尸体之间不相互推挤（避免模拟级联）。
- 尸体在可调 lifetime 后淡出（默认 60 s，实体预算紧张时可提前）。

## 12. 验证

一条规范被视为已验证当且仅当：

1. 数值目标已落入代码或 `feel_tunables.gd`。
2. 验证日志中有简短主观 playtest 备注，确认*意图*被满足。
3. 两者可独立 diff——调数值不会偷偷改动意图契约。

验证日志位于 `docs/specs/11-feel-verification-log.md`（实施开始时创建）。
